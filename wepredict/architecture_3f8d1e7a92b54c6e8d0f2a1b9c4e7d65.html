Platform Overview

WePredict is a fully independent, AI-native speculative prediction market exchange by WebPrefer. Players trade YES/NO shares on real-world events peer-to-peer through an internal Central Limit Order Book (CLOB). External feeds (Polymarket, Kalshi, Manifold, Augur) are demoted to price reference and liquidity backstop — used only when no internal counterparty exists.

A full suite of autonomous AI agents drives market analysis, order routing, fraud detection, settlement verification, and personalised recommendations. WePredict has its own users, auth, wallet, and back-office, and exposes a standard Operator Integration API so external platforms (PAM or any other operator) can embed WePredict as a prediction market provider.

What makes WePredict different 63% of fills match peer-to-peer internally — players get true exchange pricing, not aggregator markups. WePredict earns only a 2% fill fee and never takes a directional position. The MarketMakerAgent seeds spread-neutral depth so every market has liquidity from day one.

What the platform does

  • Exchange — Internal CLOB matches YES buyers against NO buyers peer-to-peer; external feeds are backstop only
  • Trade — Buy YES/NO on real-world events. Limit and market orders, GTC/IOC/FOK, live depth chart, partial fills, position close at any time
  • Provide liquidity — MarketMakerAgent quotes spread-neutral bid/ask around AI fair value so every market has depth from day one
  • Automate — AI trading bots execute strategies autonomously within user-set spend limits, with plain-English explanation for every trade
  • Analyse — AI agents compute fair value, aggregate news + social sentiment, flag mispriced markets and execution-quality anomalies in real time
  • Protect — Fraud detection (spoofing, layering, wash trading), risk monitoring, and dual-source settlement verification running continuously
  • Converse — Natural language trading assistant: "Put $100 on Brazil winning the World Cup" or "Close half my Fed rate cut position"
  • Extend — FeedBuilderAgent reads any exchange's API docs and generates a new adapter automatically

⚙ Exchange Mechanics Non-negotiable invariants

Read this before touching OrderEngine, OrderMatcher, PositionLedger, WalletService, MarketMakerAgent, or LiquidityBackstop These rules are not preferences — they are the financial integrity of the platform. Violations have caused real bugs.

The zero-sum invariant

Every binary market has exactly two sides — YES and NO. Every share pair always totals exactly $1.00:

YES buyer pays X¢  +  NO buyer pays (100 − X)¢  =  100¢  =  $1.00 per share

At resolution the winner receives $1.00 per share, the loser receives $0. WePredict holds the full $1.00 in escrow per share pair and pays it out. This is not a fee model or house-edge model — it is pure intermediation.

WePredict's revenue model

WePredict earns 2% of fill volume at the time of each match. That is the only revenue source from trading.

WePredict revenue per fill = filledShares × fillPrice × 0.02
WePredict never profits or loses from market outcomes It does not hold YES or NO positions on its own behalf. The only exception is the MarketMakerAgent, which holds temporary positions designed to net to zero. Any code path that results in the platform accumulating a directional position outside of MM is a bug.

The three-layer fill model — priority order is strict

L1
Internal CLOB (peer-to-peer) — User A's YES order matches User B's resting NO order at compatible price. WePredict is not a counterparty — it is the marketplace. Both wallets debited at placement, positions credited on fill. WePredict collects the 2% fee.
≥ 60%
target
L2
MarketMakerAgent (platform quotes) — When no resting counterparty exists, the MM places limit orders using the platform pool wallet around MarketAnalystAgent fair value. Quotes are spread-neutral by design: the MM buys/sells both sides as flow arrives and nets exposure to zero. Earns the spread, not directional P&L.
fills the gap

WePredict pool wallet executes on the external venue and holds the position on the player's behalf. External feed fee absorbed by platform — player always pays 2%.

Position netting — selling = placing the opposite side

WePredict does not have a "sell" order type. To exit a position, a player places an order on the opposite side:

Player holds: YES 100 shares at 12¢ entry
Player wants to exit: places NO order for 100 shares (market order)

If NO order fills at 17¢ bid:
  Player receives 100 × 17¢ = $17.00
  Position nets to zero (100 YES − 100 NO = 0)
  RealisedPnl = 100 × (17¢ − 12¢) = $5.00

PositionLedger handles netting automatically:
  - Same-side fill:     increase shares, update volume-weighted avg entry price
  - Opposite-side fill: reduce shares (partial close) or delete position (full close)
  - Overclose fill:     flip position side, update entry price to fill price
Never allow a player to hold simultaneous YES and NO positions in the same market PositionLedger must net them on every fill. If a player has 100 YES and 60 NO at the same time — that is a PositionLedger bug, not a valid state.

Interface-first BL services

Every BL service lives behind an interface, even if only one implementation exists today. The two-file cost buys testability, mockability, and an explicit contract.

// ✓ correct
public interface IPlayerAdminService { Task<PlayerListResult> Search(...); }
internal sealed class PlayerAdminService : IPlayerAdminService { ... }

// ✗ wrong — concrete class registered and injected directly
internal sealed class PlayerAdminService { ... }

EF Core query discipline

  • Always use AsNoTracking() for read-only queries
  • No N+1 queries — use .Include() or projection with .Select()
  • No .ToList() without a preceding .Where() — never load an entire table
  • Use CountAsync(predicate) directly — not .Where(...).Count()
  • Projection over hydration — .Select(x => new Dto(...)) when only a few columns are needed
Memory discipline — non-negotiable The Markets, TradeOrders, PriceSnapshots, EventAuditLogs, Positions, and WalletTransactions tables can grow into the hundreds of thousands of rows. Every ToListAsync() on these tables MUST have .Take(N) in the same chain. Cached endpoints must cap before the cache is built — otherwise the first request that populates the cache still loads the unbounded result set. Aggregate in SQL, not in C# (GroupBy in SQL, not .ToListAsync().GroupBy()).

Result types over exceptions for expected failures

Business failures that are part of normal operation (insufficient balance, user not found, duplicate transaction) use typed result objects, not exceptions. Exceptions are for unexpected failures only.

// ✓ caller can inspect and handle all outcomes
public sealed record OrderResult(bool Success, string? Error = null, string? OrderId = null);

// ✗ exception for a normal business outcome
throw new InsufficientBalanceException("Balance too low");

DTOs & value objects: sealed record

All data transfer objects, API request/response shapes, and internal value objects are sealed record — not class. Free value equality, immutability, and clarity that the type is a data carrier, not a service.

Cancellation token discipline

Every async method that performs I/O accepts CancellationToken ct = default and passes it through — BL methods, controller actions, Razor OnInitializedAsync, agent methods. Ignoring cancellation holds threads during request teardown.

Token governance — non-negotiable

Every LLM call costs money and burns quota Treat tokens like database writes — only do them when there is real work to do. Agents must react to domain events; not poll on a timer unless the work is inherently periodic.
Trigger typeCorrect patternForbidden pattern
Something happened (order placed, feed broke)Subscribe to the domain eventLoop every N seconds and query DB
Periodic batch (news fetch, analyst, recommendations)Timer publishes an event → agent subscribesAgent runs LLM unconditionally every tick
On-demand (admin trigger, chat)HTTP request → single LLM callProactive polling

Required guards before every LLM call

// ✓ correct — skip LLM when no data
if (recentOrders.Count < VelocityThreshold) return "Skipped";

// ✓ correct — skip when no API configured
if (string.IsNullOrEmpty(_newsApiUrl)) return;

// ✓ correct — skip when error is null (feed is healthy)
if (string.IsNullOrWhiteSpace(error)) return;

// ✗ wrong — calls LLM unconditionally
var response = await RunAsync(...);

Other token rules

  • N-multiplier agents are high-risk: any loop over users/markets/strategies must have a meaningful filter; document worst-case call count; consider batching
  • Background agents disabled by default: AI:AgentsEnabled defaults to false; enabled per-environment in config
  • Retry policy: Polly handles transient network errors only. Quota errors (Google.GenAI.ClientError etc.) are NOT retried — retrying multiplies consumption by 4×
  • Activity log: every agent invocation calls AgentHealthTracker.RecordActivity() — written to AgentRun table, visible in Admin "Agent Monitor"

Settlement & Resolution

Never settle without dual-source agreement A single source can be wrong, hacked, or manipulated. SettlementVerificationAgent requires two or more independent sources to agree. Disagreement → market stays in pending queue until resolved (manual override possible from Admin with audit trail).

  For every Position where Side = YES:
    Payout = Position.Shares × $1.00

Auth & Security

Three auth surfaces, one platform Player cookie auth (_wp.auth) for Web. Operator API key (X-Api-Key) for the Integration API. MetaMask wallet signature for the Web3 track.

Player cookie auth

User fills login/register form on WePredict.Web (/login or /register)
    ↓ POST credentials to /api/auth/login
WePredict validates credentials against own user store
    ↓ issues encrypted ASP.NET Core auth cookie (_wp.auth)
    ↓ HttpOnly, Secure, SameSite=Strict
Blazor Server reads cookie on every circuit connection automatically
REST API /api/* controllers — same cookie, zero extra config
SignalR /hubs/* — same cookie, no extra work
[Authorize] attributes work natively throughout

WePredict.Admin uses the same auth system on its own domain — requires Operator role claim.

JWT is NOT used for user auth. Cookie auth integrates natively with Blazor Server's [Authorize] attributes and works on the same host across Blazor pages, REST API, and SignalR with no extra configuration.

Operator X-Api-Key

  • Issued and rotatable per operator in the Admin portal
  • Stored hashed in Operator table
  • One active key at a time; rotation immediately invalidates the old
  • Same key authenticates inbound calls AND outbound webhook deliveries
  • Rate-limited per operator (see Integration tab)

Web3 wallet auth

1

3

PII discipline

  • Bank details on WithdrawalRequest are encrypted at rest
  • AI agents that reach external LLMs never receive raw PII — sanitization layer tokenizes player names/emails before LLM calls
  • Operator-linked players: PII ownership stays with the operator; WePredict stores only what's necessary to settle and report

Fraud detection layer

FraudDetectionAgent monitors continuously for:

  • Spoofing — large order cancelled < 30 s after placement (intent to move price without filling)
  • Layering — ≥ 3 orders stacked at adjacent price levels by same player on same side
  • Wash trading — same player on both sides of a fill (via linked accounts)

Alerts published as FraudAlertTriggered → Admin Fraud queue. RiskMonitorAgent can escalate to automatic account suspension with AccountSuspended webhook fired to operator.

  • Token discipline — AI agents react to specific events rather than polling DB on a timer
