1 System Overview

The L3M platform is a live streaming and casino entertainment platform built as four independent services that communicate over HTTP/REST and WebSocket APIs. The platform enables real-time live streams, simultaneous multi-platform distribution, adaptive bitrate video delivery, and a real-time live chat experience.

ℹ️
AWS-hosted Streaming: The live streaming server (casinogambitz-live) is deployed on AWS ECS. This ensures low-latency RTMP ingest, fast HLS segment delivery, and minimized push-latency to AWS IVS — all running on AWS backbone networking.
Service Repository Purpose Runtime
Main App casinogambitz-app Next.js frontend + Express.js API Node.js / TypeScript
Live Streaming Server casinogambitz-live RTMP ingest, HLS output, recording, encoding nginx-rtmp + Node.js / AWS ECS
CMS casinogambitz-cms WordPress/Bedrock content & media management PHP 7.4 / WordPress 5.8
Game Server CasinoGambitz.GameServer Casino game mechanics C# / .NET / ASP.NET Core

4 Live Streaming Architecture

RTMP HLS DASH AWS ECS FFmpeg nginx-rtmp

The live streaming pipeline is built on nginx with the RTMP module, running inside a Docker container on AWS ECS. AWS hosting provides the backbone network needed for low-latency RTMP ingestion and fast global HLS delivery, as well as minimal push-latency to AWS IVS.

4.1 RTMP Ingestion

Streamers connect with any standard encoder (OBS, Streamlabs, vMix, etc.):

SettingValue
RTMP URLrtmp://live.casinogambitz.com:1935/show
Stream Key{STREAM_TOKEN} (from env)
ProtocolRTMP on TCP port 1935
Auth callbackGET /api/auth?name={key} (before accepting stream)

On receiving an RTMP connection, nginx-rtmp:

  1. Calls the auth callback to validate the stream key
  2. Begins HLS segmentation to /stream/hls/
  3. Starts recording to /stream/records/{name}.flv
  4. Pushes the stream to all configured re-stream destinations simultaneously
nginx.confrtmp {
  server {
    listen 1935;
    chunk_size 4096;

    application show {
      live on;

      # Authenticate before accepting stream
      on_publish  http://localhost:8081/api/auth;
      on_done     http://localhost:8081/api/recorded;

      # HLS output
      hls on;
      hls_path /stream/hls;
      hls_fragment 1s;
      hls_playlist_length 10s;

      # Full stream recording
      record all;
      record_path /stream/records;
      record_suffix _%Y%m%d_%H%M%S.flv;
      on_record_done http://localhost:8081/api/recorded;

      # Re-stream to all platforms simultaneously
      push rtmp://127.0.0.1:1937/app/${AWS_IVS_KEY};
      push rtmp://atl.contribute.live-video.net/app/${TWITCH_KEY};
      push rtmp://a.rtmp.youtube.com/live2/${YOUTUBE_KEY};
      push rtmp://127.0.0.1:1936/rtmp/${FACEBOOK_KEY};  # via Stunnel (TLS)
    }
  }
}

4.4 Stream Authentication

Before nginx-rtmp accepts an incoming RTMP stream, it makes an HTTP callback to the Node.js server. If the token is invalid, the stream is rejected immediately. 

nginx → GET http://localhost:8081/api/auth?name={streamKey}

authApiController.ts:
  · Validates streamKey === STREAM_TOKEN (env var)
  · Returns 200 OK     → nginx accepts stream
  · Returns 4xx        → nginx rejects stream

RTMP statistics are also available (protected by the same token):

GET http://live.casinogambitz.com/stat?token={STREAM_TOKEN}

4.5 Recording & VOD Pipeline

After a stream ends, the recorded .flv file is encoded to MP4 and automatically published as a WordPress post with video embedded. 

  Stream ends
      │
      │  nginx calls  on_record_done
      ▼
  GET /api/recorded?name={name}&path={flvPath}
      │
      ├──  Wait 2 hours  (ensures file is fully written to disk)
      │
      ├──  FFmpeg encode:  .flv  →  .mp4
      │    ·  Video codec:  libx264
      │    ·  Audio codec:  AAC
      │    ·  Preset:       medium  ·  CRF: 31
      │    ·  Output:       /stream/records/{name}.mp4
      │
      ├──  POST /wp-json/custom/media        Upload MP4 to WordPress
      │
      ├──  POST /wp-json/custom/media        Upload preview image (keyframe)
      │
      ├──  POST /wp-json/custom/posts        Create WordPress post
      │    ·  Category: 440 (video)
      │    ·  ACF: video_url, main_image
      │    ·  Status: private → published
      │
      ├──  S3 Upload  (awsHelper.ts)
      │    ·  Bucket: casinogambitz-cms-test-assets
      │    ·  Path:   records/YYMMDD/{filename}
      │    ·  ACL:    public-read
      │    ·  Cache:  max-age=31536000, immutable
      │
      └──  Cleanup:  delete .flv  ·  .mp4  ·  preview files

5 Live Chat Architecture

socket.io 4.0 tmi.js 1.8 DOMPurify emoji-mart

The live chat system combines Socket.io (platform-internal WebSocket) and tmi.js (Twitch IRC protocol) with a shared message processing pipeline. The chat UI is embedded directly in the video player as an overlay panel.

5.1 Message Processing Pipeline

All messages — regardless of origin — pass through chatUtil.tsx:

  Raw message input
        │
        ├── DOMPurify sanitize
        │   Allowed HTML tags:  <img>  <a>  <b>
        │
        ├── Profanity filter  (bad-words package)
        │   Automatically replaces flagged words with ***
        │
        ├── ASCII emoticon → Unicode emoji conversion
        │     :)  →  🙂    ;)  →  😉    :D  →  😁
        │     :P  →  😛    8)  →  😎
        │
        └── Autolinker
            Detects URLs and wraps them in  <a>  tags

5.2 Chat Transport Layer

TransportLibraryVersionPurpose
WebSocket socket.io 4.0.0 Internal platform chat (bi-directional, room-based)
Twitch IRC tmi.js 1.8.5 Reads Twitch channel chat and bridges to platform
Emoji Picker UI emoji-mart 3.0.1 Emoji picker component in chat toolbar

5.3 Video Player Chat Integration

The chat is embedded directly inside the video player as an overlay via a custom Video.js plugin:

  [...page].tsx  (Next.js page)
      └──  PlayerWithoutIVS  (@eggspot/core)
              └──  Video.js  (videojs)
                      ├──  @videojs/http-streaming     ← HLS / DASH adaptive playback
                      ├──  @eggspot/vjs-chatbox-button ← Chat overlay panel
                      └──  @eggspot/vjs-theatre-mode   ← Theatre / fullscreen toggle
💬
The @eggspot/vjs-chatbox-button plugin renders the chat panel as an overlay on the video player, enabling viewers to watch and chat simultaneously without leaving the stream page.

6 AWS Infrastructure

🚀
Why AWS for Live Streaming? The casinogambitz-live service runs on AWS ECS to leverage AWS backbone networking for low-latency RTMP ingestion, fast HLS segment delivery globally, and co-location with AWS IVS to minimize re-stream push latency. The Network Load Balancer handles RTMP (port 1935) and HTTP (80/443) with high throughput.

6.1 AWS Services Used

🖥️
ECS
Runs the nginx-rtmp + Node.js container. Staging uses on-demand tasks; Production uses a persistent service.
🪣
S3
Stores encoded VOD files and media assets with public-read ACL and long-lived cache headers.
📺
IVS
Interactive Video Service. Primary CDN re-stream target for platform viewers.
⚖️
Network Load Balancer
Exposes port 1935 (RTMP) and port 80/443 (HLS/HTTP) with high-throughput TCP handling.

11 Data Flow Diagrams

11.1 Live Stream End-to-End

  1.  Streamer publishes RTMP
      └──  OBS  →  rtmp://live.casinogambitz.com:1935/show/{key}

  2.  nginx-rtmp authenticates
      └──  GET /api/auth  →  200 OK  (or 401 stream refused)

  3.  Stream goes live simultaneously
      ├──  HLS segments  →  /hls/{name}.m3u8  (for viewers)
      ├──  RTMP push     →  AWS IVS
      ├──  RTMP push     →  Twitch
      ├──  RTMP push     →  YouTube
      └──  RTMP push     →  Facebook  (via Stunnel TLS)

  4.  Viewer loads page
      └──  Browser  →  Next.js /[...page]
              └──  PlayerWithoutIVS
                      └──  Video.js  +  HLS.js
                              └──  GET /hls/{name}.m3u8
                                      └──  GET /hls/{name}_{quality}.ts  (segments)

  5.  Live chat is active
      ├──  socket.io WebSocket  →  platform internal chat
      └──  tmi.js               →  Twitch chat bridge

11.2 VOD Processing Flow

  Stream ends  →  nginx calls on_record_done
      │
      ▼
  GET /api/recorded?name={name}&path={flvPath}
      │
      ├── [ 2-hour delay ]
      │
      ├── FFmpeg:  .flv  →  .mp4  (H.264 / AAC)
      │
      ├── POST /wp-json/custom/media     ← upload MP4
      ├── POST /wp-json/custom/media     ← upload preview image
      ├── POST /wp-json/custom/posts     ← create video post (category 440)
      │
      ├── S3 upload  →  records/YYMMDD/{filename}
      │
      └── Cleanup:  delete local .flv · .mp4 · preview
